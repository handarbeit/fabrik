# Fabrik ‚Äî Development Guide for Claude

## Project Overview

Fabrik is a Go CLI that orchestrates Claude Code through an SDLC pipeline defined on a GitHub Project board. Issues are the unit of work. The pipeline stages (Research ‚Üí Plan ‚Üí Implement ‚Üí Review ‚Üí Validate) are configured via YAML files.

## Build & Test

```bash
go build -o fabrik .     # Build
go test ./...            # Run all tests
go test -race ./...      # Run with race detector
go vet ./...             # Lint
```

## Architecture

- `cmd/root.go` ‚Äî CLI entry point, flag parsing, .env loading
- `engine/engine.go` ‚Äî Engine struct, Config, construction, Run() entry point
- `engine/poll.go` ‚Äî Main poll loop, idle-upgrade, concurrent worker dispatch
- `engine/item.go` ‚Äî Per-issue processing: stage runs, comment processing, blocking/pausing
- `engine/pr.go` ‚Äî Output posting: issue comments, PR comments, summary extraction
- `engine/comments.go` ‚Äî Comment detection and filtering logic
- `engine/context.go` ‚Äî Context files (.fabrik-context/) and stage comment lookup
- `engine/repo.go` ‚Äî Per-repo identity helpers (parseOwnerRepo, repoName, issueKey)
- `engine/claude.go` ‚Äî Claude Code invocation, prompt building, marker extraction
- `engine/worktree.go` ‚Äî Git worktree lifecycle (create, update, push, cleanup)
- `engine/interfaces.go` ‚Äî GitHubClient and ClaudeInvoker interfaces (for testing)
- `github/project.go` ‚Äî GraphQL board fetching (single query for all items + comments + linked PRs)
- `github/client.go` ‚Äî HTTP client construction and shared request helpers
- `github/labels.go` ‚Äî Label mutations (add, remove, ensure)
- `github/comments.go` ‚Äî Comment mutations (add, update, reactions)
- `github/prs.go` ‚Äî PR mutations (create draft, mark ready)
- `github/status.go` ‚Äî Project board status updates
- `github/rest.go` ‚Äî Low-level HTTP helpers
- `github/types.go` ‚Äî Shared data types (ProjectItem, Comment, ReactionGroup)
- `stages/stages.go` ‚Äî YAML stage config loading
- `stages/examples/` ‚Äî Default stage YAML sources, embedded in binary via `//go:embed`
- `stages/embed.go` ‚Äî Exposes embedded default stages as `stages.DefaultStages`
- `plugin/embed.go` ‚Äî `FabrikPlugin` embed.FS; source of truth for all built-in skills
- `plugin/refresh.go` ‚Äî `RefreshPlugin()` overwrites `.fabrik/plugin/` from the embedded source
- `cmd/init.go` ‚Äî `fabrik init` subcommand; extracts embedded YAMLs to `.fabrik/stages/`
- `.fabrik/stages/` ‚Äî Live stage configs for this project (tracked in git)

> **Editing built-in skills**: modify `plugin/fabrik-plugin/skills/<name>/SKILL.md` ‚Äî the embedded source baked into the binary. `.fabrik/plugin/` is the local deployed copy written by `fabrik init` and refreshed by `fabrik upgrade`; it is not tracked in git and edits there will be silently overwritten on the next refresh.

## Key Patterns

### Reaction Flow
- üëÄ (eyes) = comment acknowledged, processing started
- üöÄ (rocket) = comment processed successfully
- The rocket reaction is checked on restart to avoid reprocessing ‚Äî it's durable state

### Markers in Claude Output
- `FABRIK_STAGE_COMPLETE` ‚Äî Claude signals stage completion (must be on its own line)
- `FABRIK_BLOCKED_ON_INPUT` ‚Äî Claude signals it needs user input before the stage can continue; mutually exclusive with `FABRIK_STAGE_COMPLETE`
- `FABRIK_ISSUE_UPDATE_BEGIN` / `FABRIK_ISSUE_UPDATE_END` ‚Äî Updated issue body from comment processing
- `FABRIK_SUMMARY_BEGIN` / `FABRIK_SUMMARY_END` ‚Äî Brief summary for issue when detailed output goes to PR

### Context Files

Before each stage invocation, the engine writes context documents to `.fabrik-context/` in the worktree:
- `.fabrik-context/issue.md` ‚Äî the issue body (spec)
- `.fabrik-context/stage-<Name>.md` ‚Äî output from prior stages
- `.fabrik-context/pr-description.md` ‚Äî linked PR description (for `post_to_pr` stages)
- `.fabrik-context/codebase-changes.md` ‚Äî files changed on `origin/<baseBranch>` since the last stage ran (generated by `context.go`; omitted when no prior stage or when SHAs match)

### Concurrency
- Workers dispatch via semaphore (`MaxConcurrent`, default 5)
- `processedSet` protected by `sync.Mutex`
- Worktree creation serialized by mutex (git config isn't concurrent-safe)
- In-flight issues tracked via `sync.Map` to prevent duplicate dispatch

### Worktrees
- Each managed repo is always bare-cloned to `.fabrik/repos/<owner>-<repo>.git` on first access
- Each issue gets `.fabrik/worktrees/<owner>-<repo>/issue-N/` on branch `fabrik/issue-N`
- `fabrikDir` (where `.fabrik/` config, stages, and plugin live) is always `os.Getwd()`
- NEVER destroy worktrees with existing content ‚Äî they may have partial work
- `updateWorktreeFromMain` fetches and merges origin/main; leaves conflicts for Claude
- Dirty worktrees (uncommitted changes) skip the update

### PR Lifecycle
- Implement creates draft PR with `Closes #N` in body (links PR to issue)
- `closedByPullRequestsReferences` in GraphQL traverses issue ‚Üí linked PRs ‚Üí PR comments
- `post_to_pr` stages post detailed output on PR, summary on issue
- PR marked ready after Implement completes (triggers external review bots)


## Important Conventions

- **Don't commit directly to main from worktrees** ‚Äî always work on the issue branch
- **Every PR must include `Closes #N`** in the body so Fabrik can discover PR comments
- **Commit frequently** during implementation ‚Äî preserves progress if session is interrupted
- **Rebase onto the latest base branch** (default branch, or the branch specified by `base:<branch>` label) in Review and Validate stages before signaling completion
- **Check `git status` first** in any stage ‚Äî there may be uncommitted work from a previous session
- **Labels are state**: `fabrik:locked:<user>`, `fabrik:editing`, `fabrik:paused`, `fabrik:awaiting-input`, `fabrik:awaiting-review`, `stage:<name>:in_progress`, `stage:<name>:complete`, `stage:<name>:failed`, `model:<name>`, `effort:<level>`, `fabrik:yolo`, `fabrik:cruise`, `fabrik:unrestricted`, `base:<branch>`
  - `model:<name>` ‚Äî set by user to select a specific model for this issue (e.g. `model:opus`)
  - `effort:<level>` ‚Äî set by user to override the stage's configured thinking effort for this issue only; valid values: `low`, `medium`, `high`, `max`; if multiple `effort:` labels are present, precedence is `max > high > medium > low`
  - `fabrik:yolo` ‚Äî set by user to force auto-advance even when `auto_advance: false` in stage YAML; also triggers auto-merge of the linked PR when Validate completes
  - `fabrik:cruise` ‚Äî set by user to auto-advance through all stages without auto-merging the PR or advancing to Done at Validate completion; if both cruise and yolo are present, yolo takes precedence
  - `fabrik:awaiting-review` ‚Äî set by engine when a stage with `wait_for_reviews: true` completes and outstanding PR reviewer requests remain; cleared when all reviewers submit or `FABRIK_REVIEW_WAIT_TIMEOUT` elapses
  - `fabrik:unrestricted` ‚Äî passes `--dangerously-skip-permissions` instead of `--permission-mode dontAsk`; bypasses the default tool allowlist entirely. Use only when a stage needs tools outside the default set (e.g. non-standard toolchains). **Caution:** removes all tool restrictions.
  - `base:<branch>` ‚Äî set by user to override the worktree base branch for this issue; Fabrik will fork from, rebase onto, and target PRs at `<branch>` instead of the repository default; must be set before Research; multiple `base:` labels use the first and logs a warning; if the branch does not exist on the remote, Fabrik falls back to the default and posts a comment

## Startup Board Validation

On every startup, Fabrik fetches the project board and compares stage names to board columns. If any non-cleanup stage is missing from the board, Fabrik exits with a detailed error message listing the mismatched names. Extra board columns (without a matching stage) produce a warning but don't block startup. This catches mismatches between stage YAML config and the GitHub Project board configuration early.

## Common Issues

- **Startup board validation failure**: Stage names in `.fabrik/stages/*.yaml` must match the column names on your GitHub Project board exactly. Check both for typos.
- **Max turns exceeded**: Increase `max_turns` in stage YAML or split the issue
- **Merge conflicts**: Left as conflict markers for Claude to resolve ‚Äî check `git status`
- **Stale worktree**: `updateWorktreeFromMain` runs on each stage invocation; skip if dirty
- **SSH key expired**: `ssh-add ~/.ssh/<key>` ‚Äî git operations fail silently with warning
- **processedSet is in-memory**: Rocket reactions provide durable "already processed" state across restarts
