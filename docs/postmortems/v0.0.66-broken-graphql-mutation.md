# Post-Mortem: Broken GraphQL Mutation Ships in v0.0.66

**Date**: 2026-05-25
**Severity**: High (silent production failure — no error, no log, feature silently no-ops)
**Fixed in**: PR #798 (v0.0.67)
**Related issue**: #799

---

## Incident Summary

Fabrik v0.0.66 shipped the cross-repo sub-issue decomposition feature with a broken
`AddBlockedByIssue` implementation in `github/project.go`. The code called a GraphQL
mutation that does not exist in GitHub's API:

| What shipped | What actually exists |
|---|---|
| `addBlockedByIssue(input: {issueId, blockedById})` | `addBlockedBy(input: {issueId, blockingIssueId})` |

When a user triggered the code path in production, the GitHub API returned an error
(`Could not resolve to a Field with the name 'addBlockedByIssue'`). The parent issue was
never linked to its spawned children as a dependency, so the blocking gate never fired and
the decomposition feature silently did not work.

The bug was introduced during the Implement stage of issue #762/#780. The spec
(`specs/sub-issue-decomposition/spec.md`) had the correct mutation name (`addBlockedBy`)
and field (`blockingIssueId`) in multiple places. The implementation diverged and was
never caught by unit tests, Review, or Validate.

The code included a comment claiming the implementation had been "Verified via schema
introspection against api.github.com/graphql on 2026-05-24." This comment was written
in good faith but the verification was performed against the read field (`Issue.blockedBy`)
— not the write mutation. The mutation name itself was never confirmed.

## Investigation

### 1. Why did unit tests not catch this?

`AddBlockedByIssue` had **no unit test** in `github/mutations_test.go`. Confirmed by
inspection: the file contained tests for `AddComment`, `AddLabelToIssue`,
`UpdateProjectItemStatus`, `FetchStatusField`, label seed operations, and `UpdateComment`
— but nothing for `AddBlockedByIssue` or `AddProjectV2ItemById`.

Even if a test had existed, the existing test pattern would not have caught the bug.
Tests in `github/*_test.go` use `httptest.NewServer` with a fake server that accepts any
request and returns whatever JSON the test specifies. The test server never validates the
GraphQL document against GitHub's real schema. The existing tests (e.g.,
`TestUpdateProjectItemStatus_Success`) check only the `variables` map — not the `query`
string. A test that only verified `vars["issueId"] == "I_child"` would pass regardless of
whether the mutation name was `addBlockedBy` or `addBlockedByIssue`.

The structural fix is for tests to additionally check that the `query` string contains the
correct mutation name and input field names. This is the approach taken in the new
`TestAddBlockedByIssue_QueryShape` test added by this issue.

### 2. Why didn't Implement notice?

The Implement stage runs `go build ./...`, `go test ./...`, and `go vet ./...` in a
sandboxed worktree. None of these tools validate that a string literal embedded in Go
source is a valid GraphQL document against a remote schema. The implementation compiled
cleanly, all tests passed (tests that existed), and `go vet` reported nothing.

The Implement stage has no mechanism to call `api.github.com/graphql` for schema
validation unless explicitly programmed to do so. "Compiles + unit tests pass" is the
expected bar for Implement — it is not a regression because calling live APIs during
implementation would be slower and would require production credentials to be available
in every Implement worktree. The gap is in the tests themselves, not in Implement's
success criteria.

The misleading verification comment compounded the problem: it gave reviewers and future
readers false confidence that the mutation had been validated against the live schema.

### 3. Why didn't Review catch the spec/implementation divergence?

The Review stage's `fabrik-review` skill prompt, under "Correctness," asked only:

> "Does the code do what the spec requires?"

This is intentionally broad, but it did not include an explicit instruction to open
the spec document and compare API-level names (mutation names, REST paths, input field
names) character-by-character against the implementation.

The spec (`specs/sub-issue-decomposition/spec.md`) used `addBlockedBy` in four separate
places (FR-010, FR-013, Key Entities, and Edge Cases). The implementation used
`addBlockedByIssue`. A Review agent reading the spec alongside the code would have caught
this — but the skill prompt did not direct the agent to perform that specific comparison.

The skill prompt was updated by this issue to add an explicit checklist item:

> If a `specs/` document exists for this feature, explicitly compare external API call
> names against the implementation: GraphQL mutation names, REST endpoint paths, input
> field names, and variable names. The spec name is authoritative — flag any divergence
> even if the code compiles and tests pass.

### 4. Why didn't Validate catch it?

Validate followed the same structural path as the other stages:

1. Ran `go test ./...` — passed, because all tests were mock-based or entirely absent for
   this function.
2. Ran `go vet ./...` — passed, because the mutation name is an opaque string.
3. Rebased onto main and confirmed the PR merged cleanly.

Validate's success criteria is "tests pass + build clean + rebased." No live API call was
in scope. This is the same structural blind spot as Implement and Review: mocked tests
pass on invalid GraphQL documents.

### 5. Contributing factor: GitHub API naming asymmetry

GitHub's API has an asymmetric naming convention for the `blockedBy` feature:
- **Read**: `Issue.blockedBy` (GraphQL field on the `Issue` object)
- **Write**: `addBlockedBy` mutation (not `addBlockedByIssue`)
- **Input field**: `blockingIssueId` (not `blockedById`)

This asymmetry between read-side and write-side names is non-obvious and easy to get
wrong. The `addBlockedByIssue` name is a natural inference from `Issue.blockedBy` — it
sounds like what the mutation should be called. This naming trap contributed to the bug.

### 6. Contributing factor: ADR naming confusion

`adrs/048-spawn-child-engine-side.md` used `addBlockedByIssue` and `addBlockedBy`
interchangeably in adjacent paragraphs — one as the Go function name and one implicitly
as the mutation name — without clearly distinguishing the two. This was not the source of
the bug, but the proximity of the two similar names in design documents may have
reinforced the confusion during implementation.

## Systemic Changes Adopted

### Change 1: Query-shape regression tests (immediate, this PR)

A new test `TestAddBlockedByIssue_QueryShape` in `github/mutations_test.go` checks that
the HTTP request body sent to GitHub contains:
- The correct mutation name `addBlockedBy` (and asserts the nonexistent
  `addBlockedByIssue` is absent)
- The correct input field `blockingIssueId` (and asserts the nonexistent `blockedById`
  is absent)
- Correct variable names in the `variables` map

This test would have caught the original bug at `go test` time, before any code review.
The existing test infrastructure (`httptest.NewServer`) supports this check with no
additional tooling or credentials — it was simply never applied.

**Going forward**, new tests for GraphQL mutations should always include a query-shape
assertion on the request body, not just variable assertions.

### Change 2: CI workflow (immediate, this PR)

`.github/workflows/ci.yml` now runs `go vet ./...` and `go test -race -timeout 5m ./...`
on every pull request targeting `main`. Previously there was no CI step for tests — only
a `docs-drift.yml` workflow for documentation changes and a `release.yml` for tags.

The `wait_for_ci: true` gate in `validate.yaml` already existed but never fired because
there was no CI workflow to produce check runs. This PR activates that gate.

**Impact**: any future PR that introduces a test regression (including a query-shape
regression test that catches a bad mutation name) will now fail CI and be blocked from
merging.

### Change 3: Review skill spec→implementation diff (immediate, this PR)

`plugin/fabrik-workflows/skills/fabrik-review/SKILL.md` was updated to add an explicit
checklist item under "Correctness" directing the Review agent to compare external API
names between the `specs/` document and the implementation whenever a spec exists.

This check catches divergences that mocked tests cannot: a mutation name can be
syntactically valid Go and pass all tests while calling a nonexistent API.

### Change 4: ADR 048 naming fix (immediate, this PR)

`adrs/048-spawn-child-engine-side.md` was updated to clearly distinguish between the Go
function name (`AddBlockedByIssue`) and the GraphQL mutation name (`addBlockedBy`) in the
rationale section that previously used them interchangeably.

## Mutation Audit

All six currently-defined GraphQL mutations in `github/*.go` were verified against the
live GitHub GraphQL schema via schema introspection on 2026-05-25.

**Verification command**:
```bash
gh api graphql \
  -f query='{ __schema { mutationType { fields { name } } } }' \
  --jq '.data.__schema.mutationType.fields[].name' \
  | grep -E "addBlockedBy|updateProjectV2ItemFieldValue|archiveProjectV2Item|markPullRequestReadyForReview|resolveReviewThread|addProjectV2ItemById"
```

```
addProjectV2ItemById
archiveProjectV2Item

resolveReviewThread
updateProjectV2ItemFieldValue
```

**Input fields for `addBlockedBy`** (verified via `__type` introspection):
```bash
gh api graphql \
  -f query='{ __type(name: "AddBlockedByInput") { inputFields { name } } }' \
  --jq '.data.__type.inputFields[].name'

```


## Recommended Follow-Ups

### F-1: Schema validation in CI (not adopted — requires PAT secret)

A CI step that introspects GitHub's GraphQL schema and validates every mutation and field
reference in `github/*.go` would catch this class of bug automatically. Tooling options:
- `graphql-inspector` — validates a `.graphql` schema against extracted queries
- A custom `go generate` step that fetches the schema and diffs against a committed copy
- A test that calls the introspection endpoint at `go test` time

**Prerequisite**: GitHub's schema introspection requires a valid PAT. A CI secret
(separate from the release token) would need to be provisioned for this workflow. The
appropriate token scope is `repo` for private repos or no scope for public repos.

This is the highest-leverage follow-up and should be tracked as a separate issue.

### F-2: Verified-API-call comment convention

Consider requiring that every GraphQL document string carry either:
- `// Verified against live schema YYYY-MM-DD` (with the mutation name and key fields
  confirmed, not just the read-side field), or
- `// Untested against live schema — see issue #N`

The comment on `AddBlockedByIssue` claimed verification but was evidently written after
verifying the read field `Issue.blockedBy`, not the write mutation. A stricter convention
would require explicitly naming what was verified:

```go
// addBlockedBy mutation; blockingIssueId input verified via __type(name: "AddBlockedByInput")
// against api.github.com/graphql on 2026-05-24.
```

This is a lightweight convention that costs one line per function and requires no tooling.
